# Pocket VM Lab

`pocket-vm-lab` 是一个刻意保持“小而完整”的玩具字节码虚拟机项目，用来测试 coding agent 在一个非 Web 场景下的持续构建能力。

这个仓库的目标不是做成下一代生产级语言运行时，而是围绕一个足够有技术含量、又能在几百到几千行代码内逐步长出来的系统，持续观察 agent 在以下问题上的表现：

- 是否能从空仓库开始搭建合理结构
- 是否能定义清晰、可扩展但不过度设计的指令集
- 是否能在解释器、汇编器、调试器、测试之间维持一致性
- 是否能在不断迭代时控制复杂度，而不是把仓库带向混乱

## 为什么这个项目适合测试 Coding Agent

和常见的 Todo、博客、仪表盘相比，虚拟机项目更适合检验 agent 的“工程推理”能力，而不是模板拼装能力。

它天然要求 agent 处理以下问题：

- 设计数据结构：寄存器、栈、常量池、程序计数器、调用帧
- 设计协议与格式：字节码编码、汇编语法、错误信息格式
- 处理语义一致性：汇编器产出的字节码必须和解释器语义严格对齐
- 写出高价值测试：不是只测 HTTP 接口，而是测执行语义、边界条件和错误路径
- 进行渐进式重构：随着功能增加，模块边界必须越来越清晰

换句话说，这个题目更容易区分“会生成代码”和“会做工程”的 agent。

## 项目目标

第一阶段，这个仓库希望完成一个最小但闭环的系统：

1. 一个简单的字节码格式
2. 一个可执行字节码的虚拟机
3. 一个把文本汇编编译成字节码的汇编器
4. 一个可以单步执行的调试模式
5. 一组足以约束语义的测试样例

后续再逐步加入：

- 函数调用
- 条件跳转
- 局部变量
- 简单标准库
- 反汇编器
- 性能分析与执行追踪

## 建议的核心能力

为了让项目保持“有趣但不过量”，建议核心能力控制在下列范围内。

### 1. 虚拟机执行模型

建议采用以下两种方案之一：

- 栈机：实现简单，适合快速起步
- 寄存器机：更接近真实虚拟机设计，能测试更复杂的指令编码

如果目标是更快进入实现阶段，优先栈机；如果目标是更好地区分 agent 的抽象能力，优先寄存器机。

### 2. 最小指令集

初始版本可以只实现这些指令类别：

- 常量加载：`CONST`
- 算术运算：`ADD`、`SUB`、`MUL`、`DIV`
- 栈/寄存器搬运：`PUSH`、`POP`、`MOV`、`DUP`
- 比较与跳转：`CMP`、`JMP`、`JZ`、`JNZ`
- 输出与停止：`PRINT`、`HALT`

这是一个足够小、但已经能写出有意义程序的集合。

### 3. 汇编器

汇编器负责把人类可读的文本转换成字节码。建议支持：

- 标签
- 注释
- 常量字面量
- 基本错误定位
- 指令参数校验

一个偏栈机风格的输入示例：

```asm
start:
  CONST 0
  CONST 10
loop:
  PRINT
  CONST 1
  ADD
  DUP
  CONST 10
  CMP
  JNZ loop
  HALT
```

### 4. 调试能力

调试器不需要一开始就很重，但至少要支持：

- 单步执行
- 查看当前 PC
- 查看栈或寄存器内容
- 查看最近执行的指令
- 程序结束状态输出

这类功能非常适合测试 agent 是否能在“已有系统上追加工具层”，而不是每次都重写主逻辑。

## 推荐技术方向

这个项目本身与具体语言无强绑定，但不同语言适合测试的能力略有不同：

- Rust：适合测试类型建模、错误处理、模块边界和性能意识
- Go：适合测试清晰实现、CLI 组织、并发扩展和工具链整合
- C#：适合测试面向对象建模、CLI、调试器组织和单元测试设计
- Zig：适合测试底层建模、内存控制和更贴近系统编程的实现

如果目标是“最稳妥地让 agent 快速做出可运行版本”，推荐 Go 或 C#。
如果目标是“看 agent 在复杂约束下是否还保持工程质量”，推荐 Rust。

## 里程碑

建议按下面的顺序推进，避免一上来把功能面铺太大。

### Milestone 0: 仓库初始化

- 选定实现语言
- 搭建 CLI 入口
- 建立测试框架
- 加入最基础的格式化和 lint 配置

### Milestone 1: 最小执行器

- 定义字节码表示
- 实现 `CONST`、`ADD`、`PRINT`、`HALT`
- 能直接加载内存中的指令序列执行
- 为每条指令补执行测试

### Milestone 2: 汇编器闭环

- 设计汇编语法
- 实现文本到字节码编译
- 支持标签解析
- 增加端到端测试：`asm -> bytecode -> vm`

### Milestone 3: 控制流

- 实现比较和跳转指令
- 支持循环程序
- 增加错误处理：非法跳转、非法操作数、栈下溢

### Milestone 4: 调试器

- 加入 `step` 模式
- 输出执行 trace
- 支持断点或至少伪断点
- 让错误信息包含当前位置和上下文

### Milestone 5: 项目打磨

- 文档化字节码格式
- 加入示例程序
- 引入反汇编器
- 评估性能瓶颈

## 可以拿来压测 Agent 的任务清单

下面这些任务很适合用来观察 agent 的真实能力，而不是仅仅看它能否一次性生成代码。

### 基础任务

- 从零初始化项目脚手架
- 实现最小指令解释器
- 写一个命令行工具来运行汇编文件
- 为每条指令补测试

### 中等复杂度任务

- 重构指令解码逻辑，避免 `switch` 失控
- 把错误类型从字符串升级成结构化错误
- 增加标签解析和前向引用支持
- 实现 trace 模式并输出执行日志

### 高价值任务

- 设计一版稳定的字节码编码格式
- 做一个简单反汇编器
- 增加交互式调试 REPL
- 设计可扩展的函数调用机制
- 在不破坏旧测试的前提下重构执行模型

### 容易暴露问题的任务

- 让 agent 追加新指令但不破坏旧指令语义
- 让 agent 同时修改汇编器、VM、文档和测试
- 让 agent 修复边界 bug 并补回归测试
- 让 agent 在已有目录结构上进行重组，而不是推倒重来

## 对 README 之外文档的建议

随着项目推进，建议补齐三类文档：

- `docs/instruction-set.md`
  明确定义每条指令的输入、输出、副作用和错误条件
- `docs/bytecode-format.md`
  记录编码格式、版本兼容策略和反汇编规则
- `docs/roadmap.md`
  记录当前里程碑、未决设计问题和下一步工作

这三份文档会显著提升 agent 多轮协作时的稳定性。

## 成功标准

如果这个项目最后能达到下面这些标准，就已经是一个很好的 agent 测试场：

- 新人打开仓库后能在几分钟内理解目标
- agent 能在没有人工重写架构的情况下连续迭代多个里程碑
- 文档、实现、测试三者长期保持一致
- 新增功能时不会频繁引入语义回归
- 仓库始终保持小而清晰，而不是演化成杂乱实验场

## 当前状态

仓库已完成 **Milestone 0** 和 **Milestone 1** 的核心部分。

### 已实现功能

- Go 模块初始化（`github.com/ponyfly6/pocket-vm-lab`）
- 最小栈式虚拟机，支持以下指令：
  - `CONST` - 将常量压入栈
  - `ADD` - 弹出两值，压入和
  - `PRINT` - 弹出并打印栈顶
  - `HALT` - 停止执行
- CLI 入口 `pocket-vm-lab`，支持 `-demo` 和 `-file` 参数
- 单元测试覆盖所有指令及错误路径

### 构建与运行

```bash
# 构建
go build -o pocket-vm-lab ./cmd/pocket-vm-lab

# 运行示例程序（计算 3 + 5 = 8，打印 8）
./pocket-vm-lab -demo

# 运行字节码文件
./pocket-vm-lab -file testdata/sample.bin
```

### 测试

```bash
# 运行所有测试
go test ./...

# 详细输出
go test ./... -v
```
